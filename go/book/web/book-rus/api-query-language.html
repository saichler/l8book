API и язык запросов

API — это язык.
Протоколы — лишь алфавит.

Поскольку API являются языками, несогласованность — не косметическая проблема; она напрямую увеличивает когнитивную нагрузку, фрагментацию и долгосрочную сложность системы.

Тем не менее в большинстве систем каждый сервис изобретает свой собственный язык, хотя все используют один и тот же алфавит.

  • Разные глаголы.
  • Разные правила.
  • Разные значения для одних и тех же концепций.

Эта фрагментация не случайна. Она является результатом отношения к API как к деталям реализации, а не как к общим коммуникационным контрактам.

Попытки нормализовать API обычно фокусируются на синтаксисе. Они стандартизируют алфавит, позволяя каждому сервису продолжать изобретать свой собственный язык.

В этом корень большей части сложности в разработке сервисов.

Layer 8 использует другой подход.

Вместо стандартизации того, как сервисы говорят, он стандартизирует то, что может быть выражено.

Эта глава описывает API межсервисного взаимодействия внутри платформы Layer 8; то, как эти API предоставляются через веб-интерфейс, рассматривается отдельно в главе Веб-сервер.

Модель как API

Как было установлено в Главе 7, сервис Layer 8 организован вокруг Первичного объекта. Сервис предоставляет фасад, следующий REST-стилю семантики, что делает естественным использование экземпляра Первичного объекта в качестве входных данных для:

POST, PUT, PATCH, DELETE и GET.

Для операций GET один экземпляр Первичного объекта действует как фильтр. Сервис возвращает все элементы, соответствующие атрибутам, предоставленным во входном экземпляре.

Это хорошо работает для отдельных взаимодействий. Однако реальные системы не работают с одним элементом за раз.

Они работают в масштабе. Масштаб требует массовых операций.

В Layer 8 входные и выходные данные вызовов API сервиса — это Elements (представленные в Главе 8).

Elements уменьшают площадь API, заменяя множество специализированных типов запросов и ответов единым унифицированным контейнером. Вместо определения отдельных операций сервиса, схем полезной нагрузки и контрактов пагинации, сервисы предоставляют небольшой фиксированный набор взаимодействий, которые все принимают и возвращают Elements.

Поскольку Elements могут представлять отдельные элементы, коллекции, частичные проекции и метаданные в одной структуре, новые возможности добавляются путём выражения намерения в модели и языке запросов, а не путём расширения самого API сервиса.

Пример: API сервиса, который полностью исчезает

В обычной микросервисной системе потребляющий сервис обычно требует специализированных операций, таких как ListEmployees, SearchEmployees или GetEmployeePage.

В Layer 8 эти операции не существуют. Вызывающая сторона выполняет:

Request(ServiceName, ServiceArea, Get, payload, timeout) -> IElements

Если payload является допустимой строкой L8Query, Elements инстанцируется как этот запрос. В противном случае Elements инстанцируется из предоставленных данных модели, выравниваясь по одному или нескольким элементам.

Ответ всегда IElements, с метаданными, описывающими область результата (например, какая страница была возвращена и сколько всего элементов существует).

Возможность заключается в модели и языке запросов, а не в новых специфичных для сервиса API.

Списки Первичных объектов

Layer 8 решает проблему больших наборов результатов, обеспечивая простое, согласованное определение списков Первичных объектов.

Для данного Первичного объекта:

message Employee {
}

соответствующий тип списка должен быть определён как:

message EmployeeList {
  repeated Employee list = 1;
  l8api.L8MetaData metadata = 2;
}

Поле list содержит запрошенную страницу результатов. Размер страницы и выбор страницы определяются L8Query.

Поле metadata заполняется автоматически инфраструктурой Layer 8. Оно может быть расширено через функции-наблюдатели для вычисления и предоставления дополнительной информации, требуемой пользовательскими интерфейсами, такой как общее количество или агрегированные значения.

Рассматривая саму модель как API, Layer 8 устраняет необходимость в пользовательских операциях сервиса, специальной логике пагинации и специфичных для сервиса форматах ответов.

Модель определяет контракт. Платформа обрабатывает механику.

Язык запросов Layer 8

Существуют языки запросов для графовых и древовидных моделей. Хотя они мощные, они часто гораздо сложнее, чем требуется Layer 8.

Во время проектирования Интроспектора (Глава 9) появилось важное понимание: поскольку Layer 8 уже понимает структуру модели во время выполнения, для запросов не требуется новый или сложный язык.

Это та же основа, что и у модельно-агностичной среды выполнения. Сервисам не нужно знание конкретных типов во время компиляции для интерпретации намерения. Платформа разрешает корневой тип модели, обходит вложенную структуру и интерпретирует пути PropertyId используя интроспекцию во время выполнения, поэтому семантика запросов остаётся стабильной даже при эволюции моделей.

Вместо этого Layer 8 использует простой, знакомый синтаксис, вдохновлённый SQL, адаптированный для работы непосредственно с вложенными моделями.

Это привело к созданию Языка запросов Layer 8.

Структура запроса

Запрос всегда начинается с ключевого слова select, за которым следует либо:

  • список атрибутов, либо
  • * для выбора полного экземпляра модели.

Атрибуты указываются с использованием нотации PropertyId, представленной в Главе 9.

Предложение from указывает корневой тип модели. В отличие от традиционного SQL, здесь нет имени таблицы. Есть ровно один корень, который представляет саму модель.

Предложение where определяет критерии фильтрации в форме:

PropertyID {оператор} значение

Пагинация обрабатывается с помощью:

  • limit для определения размера страницы
  • page для выбора номера страницы

Пример модели

message Employee {
  string id = 1;
  string name = 2;
  repeated Addr addresses = 3;
}

message Addr {
  string line1 = 1;
  string line2 = 2;
  int32 zip = 3;
}

Примеры запросов

select * from Employee

← получить все полные экземпляры модели Employee.

select id,name from Employee

← получить все экземпляры Employee только с заполненными id и name.

select * from Employee where id='layer8'

← получить полную модель Employee с id равным 'layer8'.

select id,name from Employee where addresses.zip=95124

← получить всех Employees с хотя бы одним адресом с zip 95124.

select id,name from Employee where addresses<5{1}>.zip=95124

← получить всех Employees, у которых второй адрес имеет zip 95124.

select * from Employee limit 50 page 3

← получить страницу 3 с размером страницы 50 (элементы 150-199, с нуля).

Язык запросов Layer 8 поддерживается как:

  • Распределённым кешем Layer 8, так и
  • ORM Layer 8 (рассматривается в Главе 12).

Полная справка по языку и спецификация доступны по адресу:
https://github.com/saichler/l8ql

Выравнивая семантику запросов непосредственно с моделью, Layer 8 устраняет необходимость в:

  • табличных абстракциях,
  • слоях преодоления объектно-реляционного несоответствия,
  • и специфичных для сервиса операциях запросов.

Модель определяет, что может быть запрошено.
Язык запросов выражает намерение.
Платформа выполняет его.

Примечание: Как Layer 8 предоставляет API через веб-интерфейс, рассматривается в главе Веб-сервер.
