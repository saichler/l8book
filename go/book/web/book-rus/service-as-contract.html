Сервис как контракт, обеспеченный конкурентностью

Выражение «как сервис» широко используется и редко понимается правильно.

Для большинства инженеров и архитекторов программного обеспечения сервисность не является первоклассной проектной задачей. Она не преподаётся в рамках учебных программ по информатике и не акцентируется в профессиональном развитии.

В результате системы часто проектируются без явного мышления сервисности.

Мышление сервисности

Мышление сервисности — это архитектурный подход, который рассматривает операционное поведение, корректность при конкурентном доступе, обработку отказов, восстановление и эволюцию как часть контракта сервиса, а не как запоздалую мысль.

Вместо того чтобы предполагать, что сервисы будут управляться извне через инструменты и процедуры, мышление сервисности требует, чтобы сервисы заранее декларировали свои гарантии, а платформа обеспечивала их выполнение.

В этой модели корректность, предсказуемость и диагностируемость являются свойствами самой архитектуры, а не обязанностями, перекладываемыми на операторов или рантайм-вмешательство.

С ростом микросервисов этот пробел стал более заметным. Хотя системы разбиваются на сервисы, Соглашения об уровне обслуживания (SLA) редко определяются, моделируются или обеспечиваются на этапе проектирования.

Это приводит к проектному провалу в первый же день.

Без SLA сервисы — это просто распределённые компоненты. Они предоставляют API, но не предоставляют гарантий.

В Layer 8 сервисность является основополагающей.

SLA — это не документация. Это не операционная запоздалая мысль. Это основной архитектурный контракт.

Layer 8 рассматривает SLA как первоклассную концепцию, обеспечиваемую платформой и используемую для достижения радикальной простоты в проектировании систем.

Что такое сервис?

Одно из самых распространённых заблуждений в современной архитектуре программного обеспечения — убеждение, что сервис — это процесс, контейнер или под.

Это не так.

То, что выполняется внутри процесса, не является сервисом. Отождествление рантайм-артефактов с сервисами — основной источник случайной сложности.

Это вынуждает архитекторов решать проблемы, которых не должно существовать изначально.

Это заблуждение напрямую ведёт к:

  • ненужной связанности,
  • хрупким стратегиям масштабирования,
  • и путанице в жизненном цикле.

Layer 8 использует другое определение.

В Layer 8 Сервис инкапсулирует жизненный цикл Первичного объекта.

Первичный объект

Первичный объект — это корень вложенной модели с независимым жизненным циклом и поддержкой создания, чтения, обновления и удаления состояния, и он определяет единицу владения конкурентностью, которая позволяет Layer 8 обеспечивать конкурентность на архитектурном уровне, а не в прикладном коде.

Если объект может быть создан, обновлён, запрошен и удалён независимо, он квалифицируется как Первичный объект и, следовательно, определяет границу сервиса.

Layer 8 принимает общепринятую семантику RESTful и использует POST, PUT, PATCH, DELETE и GET как базовый интерфейс взаимодействия между другими сервисами и его Первичным объектом.

Важно отметить: Этот интерфейс сервиса не открыт для внешнего мира. Он используется только для взаимодействия сервис-сервис внутри одной виртуальной сети.

Логические сервисы и рантайм-артефакты

В Layer 8 сервис — это логическая конструкция, а не рантайм-артефакт.

Логический сервис определяется владением Первичным объектом, своей моделью идентификации и декларированным Соглашением об уровне обслуживания. Он представляет контракт: как система ведёт себя при конкурентном доступе, отказах и эволюции.

Процессы, контейнеры и поды — это рантайм-артефакты. Они являются исполнительными единицами, созданными для размещения логики сервиса, но не являются самим сервисом. Они могут быть перезапущены, заменены, перепланированы или масштабированы без изменения идентичности или гарантий сервиса.

Смешение рантайм-артефактов с сервисами вынуждает архитекторов решать проблемы жизненного цикла, масштабирования и конкурентности на инфраструктурном уровне. Layer 8 избегает этого, рассматривая рантайм-артефакты как взаимозаменяемых участников единого логического сервиса, поведение которого определяется контрактом, а не развёртыванием.

Пример: один сервис

В следующем примере существует только один сервис: Employee.

Address не имеет независимого жизненного цикла. Он не существует вне модели Employee и, следовательно, не определяет собственный сервис.

message Employee {
  string id = 1;
  Address addr = 2;
}

message Address {
  string line1 = 1;
  string line2 = 2;
  string line3 = 3;
}

Пример: два сервиса

В следующем примере Address имеет собственный жизненный цикл. Он может быть создан, обновлён и удалён независимо.

В результате он становится собственным сервисом.

message Employee {
  string id = 1;
  string addrId = 2;
}

message Address {
  string addrId = 1;
  string line1 = 2;
  string line2 = 3;
  string line3 = 4;
}

Этот метод идентификации сервисов на основе владения жизненным циклом зарекомендовал себя как серьёзный архитектурный инструмент.

Он позволяет Layer 8 предоставлять обобщённую поддержку для:

  • конкурентности,
  • избыточности,
  • высокой доступности,
  • и горизонтального масштабирования,

без введения логики, специфичной для сервиса, при этом позволяя одному сервису охватывать несколько процессов и представлять себя как единый логический сервис.

Эти свойства естественно вытекают из определения сервиса и будут подробнее рассмотрены далее в этой главе.

Конкурентность и транзакции

Ключевой шаг при проектировании сервиса — определение его Соглашения об уровне обслуживания (SLA).

Если этот шаг пропущен, отложен или рассматривается как запоздалая мысль, провал практически гарантирован.

SLA — это бизнес-контракт. Если ожидания не определены заранее, потребители определят их сами, и эти ожидания, как правило, будут расти без ограничений.

Чтобы предотвратить это, Layer 8 обеспечивает осведомлённость об SLA.

SLA не является необязательным. Это обязательный входной параметр для активации сервиса.

Сервис не может запуститься без декларации своего поведения.

Без состояния или с состоянием

Первое решение, которое должен принять сервис, — является ли он без состояния или с состоянием. Допустим, три экземпляра сервиса работают в трёх разных процессах.

Если один экземпляр аварийно завершается и перезапускается, задайте следующий вопрос:

Может ли перезапущенный экземпляр немедленно обслуживать запросы без синхронизации с двумя другими и без нарушения корректности?

Если ответ — да, сервис является без состояния.

Если ответ — нет, сервис является с состоянием.

Это решение напрямую влияет на конкурентность, восстановление и семантику транзакций.

Элемент сервиса

Ранее мы установили, что сервис инкапсулирует Первичный объект.

Следующий вопрос: что уникально идентифицирует экземпляр этого Первичного объекта?

Внутри Первичного объекта один или несколько атрибутов должны определять идентичность.

Эти атрибуты формируют ключ Элемента сервиса.

Конкурентность, маршрутизация и транзакционные гарантии применяются относительно этой идентичности.

Методы конкурентности для сервисов с состоянием

Для сервисов с состоянием SLA должен определять модель конкурентности.

Layer 8 поддерживает следующие паттерны.

Наилучшее усилие

Несколько экземпляров могут получать запросы.

Когда экземпляр обрабатывает запрос, он подтверждает завершение вызывающей стороне и прилагает наилучшее усилие для распространения обновления среди своих коллег.

Эта модель отдаёт предпочтение доступности и низкой задержке, но допускает временную несогласованность.

Транзакционный

Один экземпляр избирается лидером.

Все запросы маршрутизируются через лидера, который координирует обновления с последователями.

Когда лидер подтверждает завершение, все коллеги уже синхронизированы.

Эта модель отдаёт предпочтение согласованности перед доступностью.

Счётчик репликации X

Один экземпляр избирается лидером.

Запросы маршрутизируются к лидеру, который координирует обновления с определённым числом последователей.

Завершение подтверждается только после того, как X коллег успешно синхронизировали данные.

Эта модель позволяет точно контролировать баланс между:

  • согласованностью,
  • доступностью,
  • и задержкой.

Конкурентность как сервис

В Layer 8 конкурентность не реализуется прикладной логикой.
Она не изобретается заново для каждого сервиса.
Она не угадывается во время выполнения.

Конкурентность декларируется как часть SLA и обеспечивается платформой.

Сервис выражает намерение. Layer 8 применяет контракт.

Конкурентность обслуживается, а не изобретается.

Что прикладной код больше не реализует

Поскольку поведение сервиса в Layer 8 декларируется через SLA и обеспечивается платформой, прикладной код больше не несёт ответственности за реализацию основных операционных задач.

В частности, прикладной код не реализует:

  • управление конкурентностью или семантику блокировок,
  • выборы лидера или отказоустойчивость лидерства,
  • координацию репликации или обработку кворума,
  • синхронизацию состояния между экземплярами,
  • когерентность кеша или защитное копирование,
  • логику повторных попыток для компенсации частичных отказов,
  • защиту идемпотентности для повторных запросов,
  • обнаружение экземпляров или осведомлённость о коллегах,
  • согласование жизненного цикла после перезапусков или аварий.

Прикладной код ограничен выражением доменного поведения в рамках единой абстракции логического сервиса.

Операционная корректность гарантируется платформой, а не реализуется заново каждым сервисом. В результате реализации сервисов остаются компактными, детерминированными и независимыми от топологии развёртывания.

Базовый сервис

По мере эволюции Layer 8 и создания всё большего числа инфраструктурных сервисов по тем же принципам проектирования стал проявляться паттерн.

Реализации сервисов повторяли друг друга.

Дальнейший анализ когерентности показал, что между сервисами большая часть кода реализации была фактически идентична.

Различались только декларированное намерение и SLA от одного сервиса к другому.

Через повторяющиеся циклы:

  • анализа когерентности,
  • рефакторинга,
  • и повторной валидации,

появился Базовый сервис Layer 8.

Базовый сервис фиксирует общее поведение в единой, обобщённой, хорошо протестированной реализации.

В результате реализация сервиса больше не требует написания логики сервиса.

Сервис определяется декларацией его SLA.

Остальное предоставляет платформа.

Распределённый кеш

В основе Базового сервиса лежит Распределённый кеш, разработанный для корректности при конкурентном доступе.

Кеш гарантирует как потокобезопасность, так и безопасную конкурентную модификацию, комбинируя две техники:

  • синхронизация на основе мьютексов,
  • и защитное клонирование.

Мьютекс чтения-записи позволяет нескольким горутинам читать одновременно, обеспечивая при этом эксклюзивный доступ для операций записи.

Однако одних мьютексов недостаточно.

Даже при правильной блокировке вызывающая сторона может модифицировать объект после его получения из кеша, незаметно повреждая общее состояние.

Для предотвращения этого кеш клонирует каждое значение:

  • один раз при сохранении,
  • и ещё раз при извлечении.

Вызывающие стороны всегда работают с независимыми копиями. Они никогда не получают прямых ссылок на кешированные данные.

Такой дизайн гарантирует, что:

  • изменения, сделанные одной горутиной, никогда не затрагивают другие,
  • общее состояние не может быть повреждено случайно,
  • и вызывающим сторонам не нужно самим управлять синхронизацией.

Результат — полная изоляция при минимальной когнитивной нагрузке.

Базовый сервис воплощает основную философию Layer 8:

  • декларируй намерение,
  • обеспечивай гарантии,
  • и устраняй случайную сложность.

Менеджер сервисов

Менеджер сервисов отвечает за жизненный цикл сервиса.

Он оркестрирует активацию, координацию и завершение работы сервиса и действует как плоскость управления, связывающая основные подсистемы Layer 8.

В частности, Менеджер сервисов координирует работу между:

  • защищённым оверлеем виртуальной сети,
  • руководящими принципами и политиками безопасности,
  • и декларированной моделью конкурентности, определённой SLA сервиса.

Менеджер сервисов поддерживает Реестр участников-коллег сервиса, отслеживая все активные экземпляры, участвующие в сервисе.

Когда это требуется SLA, он выполняет выборы лидера и управляет переходами лидерства прозрачно.

Лидерство рассматривается как роль, а не как идентичность, и может перемещаться без изменения фасада сервиса.

Благодаря этому механизму сервисы остаются:

  • логически единственными,
  • операционно распределёнными,
  • и соответствующими своим декларированным гарантиям.

Менеджер сервисов обеспечивает, что поведение сервиса следует намерению, а не артефактам развёртывания.

Сервис индекса репликации

Когда сервис декларирует шардинг или счётчик репликации как часть своего SLA, Layer 8 предоставляет встроенный Сервис индекса репликации.

Этот сервис отслеживает и координирует репликацию в соответствии с гарантиями, определёнными SLA.

Когда репликация включена, Сервис индекса репликации активируется автоматически.

Он поддерживает согласованное представление о:

  • том, какие экземпляры сервиса содержат реплики,
  • сколько реплик существует для каждого Элемента сервиса,
  • и как распределены обязанности по репликации.

Сервис индекса репликации работает в координации с:

  • Менеджером сервисов,
  • декларированной моделью конкурентности,
  • и защищённым оверлеем виртуальной сети.

Состояние репликации не выводится. Оно явно отслеживается и обеспечивается.
