Веб-сервер

В индустрии нет недостатка в веб-серверах. Так зачем создавать ещё один?

Потому что Layer 8 требует абсолютной когерентности и нулевой внешней зависимости.

Традиционные веб-серверы проектируются вокруг статических эндпоинтов, предопределённых маршрутов и тесной связи между транспортом, безопасностью и логикой приложения. Они предполагают, что API известны заранее и подключены вручную.

Layer 8 делает другое предположение.

В Layer 8 сервисы определяются намерением, а не эндпоинтами. API обнаруживаются, обеспечиваются и предоставляются динамически во время выполнения.

Поэтому веб-сервер Layer 8 — это не универсальный HTTP-движок. Это первоклассный архитектурный компонент.

Он интегрируется напрямую с Провайдером безопасности и системой AAA, позволяя сервисам динамически предоставлять свои API без предварительно заданных маршрутов, без ручного подключения и без утечки транспортных проблем в дизайн сервиса.

Эндпоинты не настраиваются. Они возникают.

Веб-сервер становится поверхностью выполнения для контрактов сервисов, а не местом, где встраивается логика приложения.

Это сохраняет когерентность на всей платформе, держит сервисы отделёнными от транспортной механики и обеспечивает согласованное применение безопасности, идентификации и авторизации — по дизайну, а не по соглашению.

Чем веб-сервер больше не владеет

В Layer 8 веб-сервер намеренно ограничен.

Он не владеет обязанностями, которые традиционно накапливаются внутри веб-фреймворков и HTTP-шлюзов.

В частности, веб-сервер больше не владеет:

  • Логикой маршрутизации
  • Поведением приложения
  • Решениями по дизайну API
  • Семантикой модели
  • Политикой безопасности
  • Состоянием сессии и владением конкурентностью
  • Порядком запуска и координацией зависимостей

Удаляя эти обязанности, веб-сервер сводится к тому, чем он должен быть:
поверхностью выполнения для декларированных контрактов.

Отсутствие состояния и масштабирование

Поскольку веб-сервер не владеет состоянием приложения, контекстом сессии и гарантиями сервиса, он без состояния по дизайну.

Всё поведение с состоянием, идентификация, непрерывность сессии, обеспечение конкурентности и гарантии жизненного цикла; принадлежит Провайдеру безопасности и среде выполнения сервиса, как декларировано в SLA сервиса.

В результате масштабирование веб-сервера чисто механическое. Экземпляры могут добавляться или удаляться в любое время без координации, ребалансировки или логики прогрева.

Высокая доступность достигается через репликацию, а не конфигурацию. Каждый экземпляр веб-сервера взаимозаменяем, потому что ничего внутри него не нужно запоминать.

Web API в SLA

В Главе 10 SLA сервиса был представлен как контракт, определяющий поведение, конкурентность и гарантии.

Один аспект был намеренно отложен: как сервис предоставляет свои возможности внешнему миру через Web API.

В Layer 8 внутренний API сервиса и его внешний Web API — это не одно и то же.

Между ними нет однозначного соответствия.

По этой причине каждый SLA сервиса включает специальный раздел, декларирующий Web API, которые сервис предоставляет.

Каждая запись Web API определяет:

  • входную модель
  • HTTP-метод
  • модель ответа

Запись представляет внешне видимый контракт, а не деталь реализации.

Проблемы UI vs гарантии сервиса

Web API — это механизм доставки, а не источник гарантий.

Проблемы UI и транспорта включают форматирование запросов, таймауты, повторные попытки, пагинацию, кеширование и соглашения HTTP-статусов.

Гарантии сервиса определяются исключительно SLA сервиса:

  • правила конкурентности, корректность жизненного цикла,
  • границы авторизации,
  • и наблюдаемое поведение под нагрузкой и при отказах.

Веб-сервер не определяет гарантии. Он переводит между HTTP и средой выполнения сервиса. Любая гарантия, на которую полагается клиент, должна исходить из SLA сервиса и самого сервиса.

Внутренний API-менеджер сервисов

Веб-сервер Layer 8 включает Внутренний API-менеджер сервисов, отвечающий за связь контрактов сервисов и веб-экспозиции.

Его роль проста и намеренна: собирать декларации Web API, определённые в каждом SLA сервиса, и подключать их к веб-серверу во время выполнения.

Сервисы не регистрируют эндпоинты вручную. Они публикуют свои декларации Web API при активации.

Аналогично, веб-сервер не предполагает, что сервисы уже существуют. Обнаружение симметрично и детерминировано.

Web API не настраиваются. Они обнаруживаются.

От запроса к сервису

Выполнение запроса в Layer 8 намеренно невзрачно.

  • Веб-сервер проверяет идентификацию через Провайдера безопасности,
  • десериализует JSON во входной protobuf,
  • переводит HTTP-метод в действие Layer 8,
  • и перенаправляет запрос целевому сервису.

Сервис выполняется под правилами, определёнными его SLA.

  • Ответ возвращается как Elements,
  • сериализуется в JSON,
  • и доставляется вызывающей стороне.

Ничто в этом потоке не специфично для приложения.

Выполнение следует контрактам. Транспорт остаётся деталью реализации.
