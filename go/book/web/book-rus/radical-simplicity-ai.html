Радикальная простота для ИИ

Мы широко использовали ИИ для укрепления как архитектуры, так и реализаций экосистемы Layer 8. В процессе мы обнаружили, что эффективность ИИ ограничена не интеллектом, а структурой.

В рамках этого процесса мы определили чёткий и явный набор правил использования ИИ — правил, призванных предотвращать галлюцинации, избегать дублирования кода, обеспечивать надлежащую абстракцию и согласовывать генерируемый код с архитектурным замыслом.

Вместо того чтобы ожидать, что пользователи будут раз за разом обучать ИИ тому, как работает Layer 8, мы встроили архитектурные рекомендации и API Layer 8 непосредственно в эти правила — чтобы вам не приходилось этого делать.

Мы также разработали UI-компоненты Layer 8, которые уже интегрированы с API Layer 8 и предзагружены с ИИ-руководствами. Эти компоненты действуют как ограждения, гарантирующие, что ИИ-генерируемый вывод остаётся согласованным, обслуживаемым и готовым к продакшену.

Наша цель — не просто чтобы ИИ писал код. Наша цель — чтобы ИИ создавал качественное программное обеспечение корпоративного класса — правильно, с первого раза.

Радикальная простота для ИИ означает доставку работающего продукта за часы вместо месяцев — без ущерба для архитектуры, корректности или долгосрочной поддерживаемости.

Прежде чем начать работу с ИИ, необходимо дать ему явные глобальные руководства по разработке программного обеспечения.

Это было одним из наиболее удивительных открытий в нашей работе. Несмотря на кажущуюся изощрённость, ИИ не надёжно применяет основополагающую дисциплину программной инженерии, если эта дисциплина не указана явно.

Для решения этой проблемы мы определили два отдельных набора правил:

  1. Общие правила разработки ПО
    Это архитектурно-агностичные принципы, применимые к любой серьёзной программной системе.
  2. Правила, специфичные для Layer 8
    Они кодируют архитектурный замысел, ограничения обслуживаемости и инварианты проекта, которые ИИ не может вывести самостоятельно.

Вместе эти правила превращают ИИ из генератора кода в ограниченного, архитектурно осведомлённого участника.

Примеры общих правил разработки ПО

Некоторые из наиболее эффективных общих правил были удивительно базовыми — но критически важными:

  • Никакого дублирования кода
    ИИ склонен к повторению. Любой второй экземпляр паттерна рассматривается как принуждающая функция для абстракции, а не копипаста.
  • Читай перед реализацией
    При расширении или интеграции с существующими компонентами ИИ обязан прочитать весь соответствующий код сначала. Допущения не допускаются.
  • Ограничения размера файла обеспечивают проектирование
    Файлы, превышающие определённый порог размера, рассматриваются как архитектурные неудачи и должны быть отрефакторены.
  • Конфигурация вместо поведения
    Файлы, специфичные для модуля, содержат только данные. Вся логика живёт в общих, переиспользуемых компонентах.

Эти правила радикально сократили разрастание кода, тихое расхождение и долгосрочные затраты на обслуживание.

Примеры правил, специфичных для Layer 8

Правила Layer 8 кодируют архитектурные ограничения, которые ИИ не может «угадать», даже при полном контексте:

  • Архитектура явна, никогда не подразумевается
    Сервисы, области сервисов, владение и границы должны быть указаны и обеспечены.
  • API сервисов — источник истины
    UI, оркестрация и автоматизация всегда соответствуют контрактам сервисов — не наоборот.
  • Эволюция модели ожидаема
    Проекты должны предполагать, что модели будут меняться, расти и переиспользоваться между модулями без нарушения поведения.
  • Обслуживаемость важнее удобства
    Любое решение, улучшающее краткосрочную скорость, но ухудшающее наблюдаемость, контроль или эволюцию, отклоняется.

Эти правила гарантируют, что ИИ-генерируемый код согласован с основными принципами Layer 8: явная архитектура, контролируемая конкурентность и долгосрочная обслуживаемость.

Почему это важно

Без правил ИИ оптимизирует локальную корректность. С правилами ИИ оптимизирует целостность системы.

Разница не является тонкой.

Встроив как общие, так и специфичные для Layer 8 руководства непосредственно в рабочий процесс ИИ, мы устранили:

  • Галлюцинированные абстракции
  • Тихий архитектурный дрейф
  • Взрывной рост модулей через копипаст
  • Недолговечные «рабочие» решения, которые ломаются при масштабировании

ИИ не заменяет архитектуру. Правила — это то, как архитектура обеспечивается — на машинной скорости.

Создание вашего первого ИИ-проекта

Разработка собственной ИИ-управляемой системы с экосистемой Layer 8 намеренно проста. Это руководство проведёт вас через создание первого проекта с нуля с использованием паттернов Layer 8 и ИИ-ассистируемой разработки.

Настройка среды разработки

1. Установка Go, wget и Docker/Docker desktop

Скачайте и установите Go для вашей платформы:
https://go.dev/dl/

Скачайте и установите Docker для Linux для вашего дистрибутива или Docker desktop с https://docs.docker.com/get-started/ если вы на windows/mac.

Если в вашей системе нет wget (утилита командной строки), пожалуйста, установите его.

2. Откройте терминал и перейдите в домашнюю директорию

cd ~

3. Распакуйте архив Go

tar -zxvf ~/Downloads/<golang file name>

4. Настройка переменных окружения

Добавьте следующее в файл конфигурации вашей оболочки (например, ~/.bashrc):

export GOROOT=~/go
export GOPATH=~/proj
export GOBIN=~/proj/bin
export PATH=~/.local/bin:~/go/bin:$GOBIN:$PATH

alias ll="ls -lt"

git config --global user.email "<your email address>"
git config --global user.name "<your name>"

export DOCKER_BUILDKIT=1

Перезагрузите оболочку:

source ~/.bashrc

5. Создание рабочего пространства для разработки

mkdir -p ~/proj/src/github.com/<your github username>

Эта директория будет содержать все ваши проекты Layer 8.

Создание и клонирование вашего проекта

1. Создайте новый репозиторий на GitHub

Создайте репозиторий по адресу:

github.com/<your github username>/<your project name>

2. Клонируйте проект локально

cd ~/proj/src/github.com/<your github username>
git clone https://github.com/<your github username>/<your project name>

3. Клонируйте эталонную реализацию Layer 8 ERP

git clone https://github.com/saichler/l8erp

Проект l8erp служит полным рабочим примером системы Layer 8.

Установка ИИ-ассистента для кодирования

Установите один из поддерживаемых ИИ-инструментов для кодирования:

Claude Code:
https://code.claude.com/docs/en/setup

или ChatGPT Codex (или эквивалентный кодирующий агент).

Эти инструменты генерируют архитектуру, код и документацию, следуя правилам Layer 8.

Создание документа требований к продукту (PRD)

1. Перейдите в ваш проект

cd ~/proj/src/github.com/<your github username>/<your project name>

2. Запустите сессию ИИ-кодирования

claude

или

codex

3. Используйте следующие промпты

Пожалуйста, скопируйте и вставьте следующие промпты для настройки правил и руководств экосистемы Layer 8 для ИИ:

Please install to your memory the following Layer 8 global rules so it will be available in every session: ./l8erp/plans/global-rules-all.md and make sure any PRD is in compliance to those rules.

The following ../l8erp/go/erp/ui/web/l8ui/GUIDE.md is a guide for the Layer 8 UI/UX generic components, please add it to your memory and use it whenever planning or implementing PRD with UI section.

An example project built using the Layer 8 ecosystem is in ../l8erp. Please follow this project structure and architecture whenever you create a PRD for a new project.

Create a PRD in the directory plans that comply with the global rules and follows the Layer 8 architecture for <your project/system idea>

ИИ сгенерирует полный PRD, согласованный с архитектурой Layer 8 и её ограничениями.

Начало реализации

После создания PRD:

  1. Следуйте фазам реализации, определённым в PRD.
  2. Позвольте ИИ-ассистенту генерировать код и структуру для каждой фазы.
  3. Убедитесь, что все сгенерированные компоненты соответствуют глобальным правилам Layer 8.
  4. Продолжайте итерации до полной реализации PRD.

С этого момента разработка становится управляемым выполнением PRD — с Layer 8, обеспечивающим архитектурную согласованность, и ИИ, ускоряющим доставку.
